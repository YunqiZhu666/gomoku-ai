## 1. 技术背景调研与文献分析

### 1.1 技术现状

#### 博弈树搜索的发展历程

棋类 AI 的研究一直是人工智能领域的经典方向。1997 年 IBM 的深蓝（Deep Blue）击败国际象棋世界冠军卡斯帕罗夫，使用的就是 Minimax 搜索 + Alpha-Beta 剪枝 + 定制评估函数。2016 年 AlphaGo 将深度学习与蒙特卡洛树搜索（MCTS）结合，击败了围棋世界冠军李世石，将博弈 AI 推向新高度。

在五子棋领域，当前主流解决方案包括：

1. **基于搜索的算法**：Minimax + Alpha-Beta 剪枝，配合优化的评估函数和走法排序，在小棋盘（9×9 或 15×15）上可以达到很高的棋力。代表实现有 Renjusolver、LibPuzzle 等。

2. **基于 MCTS 的算法**：蒙特卡洛树搜索通过随机模拟来评估局面，不需要手工设计的评估函数，但需要大量的模拟次数来保证准确性。

3. **基于深度学习的方法**：使用卷积神经网络（CNN）学习棋型特征和评估函数。如 AlphaGo Zero 的方法需要大量自我对弈训练。

本项目选择 **Minimax + Alpha-Beta 剪枝 + 迭代加深** 的技术路线，理由是：
- 对于 9×9 棋盘，搜索复杂度可控
- 不需要深度学习所需的 GPU 资源
- 算法原理清晰，适合课程项目展示

#### Minimax 算法原理

Minimax 是一种在零和博弈中使用的决策算法。核心思想是：

- **MAX 方**（我方）希望最大化局面分数
- **MIN 方**（对手）希望最小化局面分数
- 双方都假设对手会走出对自己最不利的走法

伪代码：
```
function minimax(node, depth, isMaximizing):
    if depth = 0 or node is terminal:
        return evaluate(node)
    if isMaximizing:
        best = -∞
        for each child of node:
            value = minimax(child, depth-1, false)
            best = max(best, value)
        return best
    else:
        best = +∞
        for each child of node:
            value = minimax(child, depth-1, true)
            best = min(best, value)
        return best
```

#### Alpha-Beta 剪枝原理

Alpha-Beta 剪枝是对 Minimax 的优化，在不改变搜索结果的前提下，通过维护两个边界值来裁剪不必要的搜索分支：

- **Alpha（α）**：MAX 方能保证的最小分数（下限）
- **Beta（β）**：MIN 方能保证的最大分数（上限）

关键观察：在搜索过程中，如果某个分支的分数已经超出对方能接受的范围（α ≥ β），则可以立即停止搜索该分支的剩余走法，因为对方不会允许局面走到这个方向。

**剪枝效率**：在走法排序最优的情况下，Alpha-Beta 剪枝可以将有效分支因子从 b 降低到约 √b，搜索深度翻倍而计算量不变。

### 1.2 核心原理预研

#### 评估函数设计

评估函数是五子棋 AI 的"棋感"来源。本项目采用基于棋型识别的评估方法：

- 沿四个方向（横、竖、对角线、反对角线）扫描棋盘
- 识别连子模式：连五→活四→冲四→活三→眠三→活二→眠二
- 为每种棋型赋予分数，综合评定局面

分值权重表：

| 棋型 | 分值 | 说明 |
|------|------|------|
| 连五 (FIVE) | 10,000,000 | 必胜 |
| 活四 (LIVE_FOUR) | 1,000,000 | 必赢（两端开放的四子） |
| 冲四 (RUSH_FOUR) | 100,000 | 一端受阻的四子 |
| 活三 (LIVE_THREE) | 100,000 | 两端开放的三子，威胁大 |
| 眠三 (SLEEP_THREE) | 10,000 | 一端受阻的三子 |
| 活二 (LIVE_TWO) | 1,000 | 潜在威胁 |
| 眠二 (SLEEP_TWO) | 100 | 微弱优势 |

最终局面分数 = 己方棋型总分 − 对方棋型总分 × 1.1（防守权重略高）

#### 走法生成优化

为了提高搜索效率，只考虑**已有棋子周围 1 格范围内**的空位作为合法走法。在 9×9 棋盘上，中期的合法走法数通常在 10-20 之间，可以保证搜索的可行性。

#### 迭代加深

在有限时间内尽可能搜索更深的层次。从 depth=1 开始逐层加深，一旦时间预算用完，立即返回当前已知的最佳走法。这样保证了在任何时候 AI 都能给出一个合理的走法。

#### 置换表（Transposition Table）

使用字典缓存已搜索过的棋盘状态及其评估结果。当不同走法顺序导致相同局面时，可以直接从缓存中读取结果，避免重复搜索。
